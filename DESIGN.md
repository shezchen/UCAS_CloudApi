# UCAS Cloud API — AxonHub 校内公益共享版设计

> 本文档描述当前仓库已经实现的架构，是后续修改与上线验收的基准。
>
> 状态：V1 代码已完成，尚未部署到生产；生产迁移、邮件投递、跨进程配额与流式硬截断仍需上线前验证。
>
> 校内发行版本：`v1.0.0-beta5-ucas.1`
> 最后更新：2026-07-20

---

## 1. 项目定位

UCAS Cloud API 是基于 AxonHub 的校内公益模型资源共享项目：

- 国科大成员用校内邮箱完成验证注册；
- 每位成员使用自己的 AxonHub Personal API Key；
- 成员可以捐赠自己的上游渠道，并自行设定有效期；
- 普通成员只能看到和管理自己的捐赠与 API Key；
- Owner 维护项目全局资源、账户日总额度和运行状态；
- 请求继续复用 AxonHub 原生协议适配、渠道能力、日志、模型和调度链路。

本项目不是公开免费代理，也不另建一套 Node/Hono 控制面，不把上游凭据或管理员能力交给普通成员。

## 2. 实际架构

本仓库直接扩展 AxonHub，而不是在它前面增加第二个网关：

```text
校内成员 / OpenAI、Anthropic、Gemini 兼容客户端
                         │
                         │ Personal API Key
                         ▼
┌─────────────────────────────────────────────────────┐
│ AxonHub 校内发行版                                   │
│                                                     │
│ Go / Gin API 与中间件                               │
│ - API Key 鉴权                                      │
│ - 用户日总额度与单 Key 额度检查                     │
│ - 每账户 4 并发                                     │
│                                                     │
│ Biz + Ent + GraphQL                                 │
│ - 校内邮箱验证注册                                  │
│ - 用户、项目与 Personal API Key                     │
│ - 捐赠渠道所有权、有效期和模型同步                  │
│ - 隐私安全用量排行榜                                │
│                                                     │
│ AxonHub 原生 LLM pipeline                           │
│ - OpenAI / Anthropic / Gemini 等协议转换            │
│ - 原生渠道凭据模式、健康状态和可用性选择            │
│ - UsageLog / Request 审计                           │
└──────────────────────────┬──────────────────────────┘
                           │
                           ▼
               捐赠者提供的上游渠道 / Coding Plan
```

前端仍是仓库内 React 管理台；后端仍是 Go 服务；数据继续存入 AxonHub 的 Ent 数据模型。这样避免重复实现协议转换、日志、渠道管理和模型目录，也降低学生管理员的维护负担。

## 3. 身份与注册

### 3.1 Owner

- 现有 Owner 账户继续是系统所有者，可查看和管理全局资源。
- 已存在的非校内邮箱 Owner（例如历史 QQ 邮箱）可以继续登录和更新其他资料；保存未改变的旧邮箱不会被新的校内域规则误拒绝。
- 如果 Owner 主动把邮箱改成新地址，新地址仍按校内邮箱规则校验。

### 3.2 校内成员注册

公开注册只接受以下精确域名：

- `mails.ucas.ac.cn`
- `mails.ucas.edu.cn`
- `ucas.ac.cn`
- `ucas.edu.cn`

邮箱会统一 trim 和转小写；不使用宽泛的后缀匹配，避免 `evil-ucas.ac.cn` 一类伪域名通过。

注册流程：

1. 用户填写校内邮箱并请求验证码；
2. 后端返回固定形状响应，避免用接口枚举已注册邮箱；
3. 验证码以摘要形式保存，有有效期、重发冷却、邮箱/IP 来源和全局频率限制、最大尝试次数；
4. SMTP 成功发送后，用户提交邮箱、验证码、密码和昵称；
5. 后端在事务中消费验证码、创建用户并加入默认项目；
6. OIDC 首次建号也受相同的精确校内域限制，并遵循提供方的邮箱验证配置。

SMTP 与验证码限制由 `campus_email_verification` 和邮件配置控制；真实 SMTP 密码只能放在运行环境或密钥管理中。

## 4. 权限与租户隔离

权限必须在后端 Ent policy、Biz service 或 resolver 中执行；前端隐藏按钮不是权限边界。

### 4.1 普通成员

普通成员只能：

- 查看、编辑、启停和删除自己捐赠的渠道；
- 查看和管理自己创建的 Personal API Key；
- 为自己的每把 API Key 设置子额度；
- 查看隐私安全的校内用量排行榜；
- 修改自己的昵称、密码等个人资料。

普通成员不能：

- 读取他人的渠道记录或上游凭据；
- 读取他人的 API Key；
- 修改自己的账户日总额度；
- 修改全局模型、模型价格、系统设置或 Owner 资源；
- 使用 Owner Playground 管理能力。

### 4.2 Owner

Owner 可以查看和管理全局资源、用户、额度和所有渠道的运行状态，但捐赠记录仍遵守公益治理约束：

- 活跃捐赠不能由 Owner 直接删除；
- 捐赠有效期不能由 Owner 擅自改写或清空；
- 捐赠者可以删除自己的捐赠；
- 到期任务可以将渠道退出运行、清除敏感凭据并软删除；
- 暂时不可用、余额耗尽或健康检查失败只影响调度，不提前删除捐赠记录。

## 5. 捐赠渠道

### 5.1 数据与生命周期

AxonHub `Channel` 增加：

- `user_id`：为空表示 Owner 的全局渠道；非空表示捐赠者；
- `expires_at`：普通成员捐赠时必须设置未来有效期。

查询、缓存、模型目录和运行选择都会排除已到期渠道。后台清理任务处理到期记录并清除敏感凭据，保留软删除和审计语义。

### 5.2 原生凭据模式

捐赠注册沿用 AxonHub 的渠道抽象，不强制所有上游都是简单 API Key。现有模式继续可用，包括：

- 普通 API Key；
- OAuth / Device Flow；
- GCP 凭据；
- Cookies；
- Coding Plan 等提供方特有凭据。

因此前端与后端都不能把渠道注册降级成单一 `api_key` 字段。

### 5.3 自定义 URL 与代理

自定义 URL 和公开代理是合法能力，不做一刀切禁用：

- 允许公开可达的 HTTP、HTTPS、WS、WSS 目标；
- 允许公开 HTTP/HTTPS 代理及代理认证信息；
- 禁止 localhost、私网、链路本地、云元数据、保留地址等内部目标；
- DNS 解析、重定向和实际 dial 都执行公共地址约束，降低 SSRF 与 DNS rebinding 风险；
- 普通成员不能借自定义 URL 访问校园内网、云主机元数据或本机服务。

Owner 管理的全局渠道保留 AxonHub 原有运维能力。

## 6. 模型目录同步与可用性

捐赠渠道会触发一次即时、尽力而为的模型同步：

- 同名模型按精确名称去重；
- 上游新模型加入渠道模型列表；
- 上游移除的自动同步模型会移出自动列表；
- 手工维护的模型不会因自动同步被误删；
- 即时同步失败时保留捐赠时提交的模型，不让一次临时网络错误毁掉渠道注册；
- 启用自动同步的渠道继续参加周期同步；
- 同步成功后立即刷新已启用渠道缓存，使查询目录与运行路由尽快一致。

模型“存在于目录”和渠道“此刻可用”是两个维度。运行时可用性继续复用 AxonHub 原生健康、错误、熔断、额度状态和候选选择逻辑；不可用渠道退出候选但记录不删除。未验证现有能力前，不再实现第二套轮询器。

## 7. Personal API Key

- 普通成员创建的 Key 强制为 `personal` 类型；
- Key 归属当前用户和当前项目；
- 默认只授予调用所需的最小 scope；
- 明文 Key 仅在创建时返回，列表不向其他普通成员暴露；
- 删除或轮换 Key 不会抹掉已经写入的 UsageLog；
- 用户可以给自己的每把 Key 设置日额度，但不能借此扩大账户日总额度。

Key 名称在项目内保持唯一，服务端在创建和更新时执行所有权与并发安全检查。

## 8. 配额与并发

### 8.1 已实现规则

- 每个用户账户日总 Token 上限默认 `200,000,000`；
- 只有 Owner 可以修改账户日总上限；
- 每把 Personal API Key 可以有用户自设的子额度；
- 请求前先检查单 Key 额度，再检查该用户所有 Key 的账户日总用量；
- 用户汇总包含软删除 Key，删除 Key 不能重置当日用量；
- 日窗口按系统配置时区的自然日计算；
- 每个用户所有 Key 合计最多 4 个在途写请求，第 5 个返回 `429 concurrency_limit_exceeded`；
- 流式响应在处理器返回前持续占用并发槽。

### 8.2 V1 明确边界

当前实现不能宣称“精确逐 token 硬截断”：

- 日额度基于请求前已持久化 UsageLog 检查；多个同时在途请求可能在最终结算时产生小幅超额；
- 4 并发计数是单进程内存状态，只适用于单实例 V1；
- 尚未实现跨实例的 Redis/数据库原子预留、结算和断连回收；
- 尚未为所有协议实现流式实时 token 计数与中途标准错误。

生产若保持单实例，可以使用当前 4 并发保护和请求前额度拒绝，但必须在用户文案中说明口径。扩展到多实例或承诺绝对硬上限前，必须加入账户级原子预留与最终 usage 结算；不能用“响应结束后统计”冒充硬截断。

## 9. 用量排行榜

普通成员看到的是单独的隐私安全 GraphQL 查询 `campusUsageLeaderboard`，不是 Owner 的详细 `usageStatsByUser`。

统计口径：

- 系统时区的今日自然日 `[start, end)`；
- 从服务端 UsageLog 按 API Key 的用户归属聚合；
- 包含正常软删除 Key 的历史用量；
- 按已记录 Token、已计量请求数排序；
- 返回 Top 50，并在本人不在 Top 50 时追加本人及真实全局名次；
- 别名由 `project + user` 的确定性哈希生成，只在项目内稳定。

公开 DTO 只包含：

- `rank`
- `publicAlias`
- `isMe`
- `recordedTokens`
- `meteredRequestCount`
- `limitPercent`

不返回 user ID、邮箱、真实昵称、成本、Key、渠道、模型、内容或 IP。页面明确写“已记录用量”和“已计量请求”；没有 UsageLog 的失败调用不应被描述成已计量请求。

Owner / 项目 Owner 仍可使用原详细统计页做运营排查，该查询没有向普通成员放开。

## 10. 数据与审计

V1 继续使用 AxonHub Ent 管理的数据库：

- `User`：Owner、状态、昵称、邮箱、账户日总额度；
- `EmailVerificationChallenge`：验证码摘要、来源摘要、有效期、尝试和消费状态；
- `UserProject`：成员关系和项目 scope；
- `APIKey`：用户、项目、类型、状态和 profile quota；
- `Channel`：捐赠者、有效期、凭据、模型和状态；
- `Request` / `UsageLog`：请求过程和服务端计量结果。

上线前必须备份生产数据库并在副本验证 Ent schema 迁移；不得为迁移方便删除现有用户、渠道、Request 或 UsageLog。

## 11. 安全基线

1. 仓库不提交 `.env`、QQ 本地笔记、日志、缓存、模型权重或真实密钥。
2. SMTP 密码、上游凭据、AxonHub Secret Key 只进运行环境或密钥管理。
3. 新注册邮箱必须完成一次性验证码验证；接口使用固定形状响应降低枚举风险。
4. 普通成员所有资源操作必须由后端校验所有权。
5. 自定义 URL 可用，但必须经过公共网络地址约束。
6. 渠道凭据只对 Owner 和对应捐赠者返回；到期后即使清理任务尚未运行也不再返回。
7. 日志和排行榜不展示 API Key、邮件、请求内容或客户端 IP。
8. 上线前轮换任何曾在群聊、截图或本地明文文件中出现过的凭据。
9. 禁止把上游套餐倒卖或扩散到公开互联网；捐赠者需确认各自套餐条款允许共享。

## 12. 生产上线门槛

V1 发布到服务器前至少完成：

- [ ] 在隔离副本上从现有生产 schema 执行迁移并验证回滚备份；
- [ ] 配置真实 SMTP，验证四个校内域的收信、限流、过期和错误文案；
- [ ] 验证历史 Owner 的登录和资料更新，确保旧邮箱不被误伤；
- [ ] 用普通成员、另一成员、Owner 三类账户验证渠道/API Key 行级隔离；
- [ ] 验证渠道即时模型同步、周期同步、缓存刷新和临时失败回退；
- [ ] 验证公开自定义 URL 可用，私网/元数据/重定向绕过被拒；
- [ ] 验证同一账户 4 个流式请求占槽，第 5 个稳定返回 429；
- [ ] 明确当前请求前额度口径；若需要绝对硬额度，先实现原子预留与流式结算；
- [ ] 验证排行榜只返回安全 DTO，软删除 Key 仍计入；
- [ ] 轮换生产凭据并检查仓库、日志和部署包中无秘密；
- [ ] 确认 `/health` 返回校内发行版本 `v1.0.0-beta5-ucas.1`；
- [ ] 全程不重启或中断受保护的 HCZ/HPC/FRP 服务。

## 13. 后续优先级

1. 跨进程账户并发和日额度原子预留 / 结算；
2. 流式 token 计量与协议一致的额度终止语义；
3. 渠道贡献榜，与消费榜分开，仍只公开匿名聚合；
4. 邮件投递、验证码与异常注册的运营监控；
5. 捐赠渠道健康与到期审计视图；
6. 在不增加管理员日常负担的前提下完善自助说明和故障自查。

任何后续设计都应优先复用 AxonHub 已有能力，避免再建一套协议层、账户库、模型目录或调度系统。
